# DEV_WORKFLOW.md — fva-analytic

## Цель
Единая последовательность безопасной разработки для AI и человека в production-репозитории.

## 1. Старт задачи
1. Уточни цель изменения и ожидаемый результат.
2. Прочитай `.ai/context.md`, затем релевантные модули.
3. Определи слой(и) архитектуры, которых касается задача.

## 2. Перед правками: impact analysis
Обязательно оцени:
- какие файлы и слои будут затронуты;
- есть ли влияние на `fva-bot` или `fva-streamlit`;
- есть ли риск косвенного влияния на deploy/webhook/systemd.

Если есть влияние на чувствительные зоны, сначала запроси подтверждение.

## 3. План до редактирования
Для multi-file или multi-layer задач сначала представь короткий план:
1. Что меняем.
2. Почему это минимально достаточный объем.
3. Какие риски и как проверяем.

Без плана не начинать сложные изменения.

## 4. Выполнение

### Bugfix
1. Локализуй источник дефекта.
2. Исправь минимально возможным диффом.
3. Проверь сценарий регрессии рядом с местом правки.

### Feature work
1. Внеси изменения в правильный слой архитектуры.
2. Не смешивай feature с реорганизацией структуры.
3. Добавь только необходимые связанные правки.

### Multi-file changes
1. Разбей на логические шаги.
2. Сначала данные/логика, затем интеграция, затем UI/бот.
3. После каждого шага проверь совместимость.

## 5. Валидация перед предложением изменений
- Проверка синтаксиса и импортов в измененных файлах.
- Локальная проверка затронутых сценариев (расчеты, загрузка, UI/бот-путь).
- Отсутствие непреднамеренных изменений в чувствительных зонах.

## 6. Формат отчета по результату
1. Что изменено (файлы и смысл).
2. Почему это безопасно для production.
3. Что проверено.
4. Какие риски/ограничения остаются.
