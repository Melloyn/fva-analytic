# AI_RULES.md — fva-analytic

## 1. Запрещено без подтверждения
- Любые изменения deploy-скриптов, webhook listener и логики автодеплоя.
- Любые изменения `systemd`-сервисов (`fva-bot`, `fva-streamlit`) и их окружения.
- Изменения инфраструктуры VPS.
- Массовые рефакторинги структуры каталогов.
- Перемещение логики между слоями, нарушающее границы архитектуры.

## 2. Разрешено
- Точечные bugfix-правки в существующих модулях.
- Локальные улучшения читаемости и надежности (минимальным диффом).
- Добавление/корректировка расчетов в `backend/analytics/`.
- Правки ingestion-процесса в `backend/ingestion/`.
- Улучшения утилит в `backend/utils/` без скрытого побочного эффекта.
- Изменения UI в `dashboard/` без вмешательства в инфраструктуру.
- Изменения логики бота в `bot/` в рамках текущей архитектуры.

## 3. Предпочтительное поведение
- Сначала понять текущую реализацию, потом менять.
- Делать малые независимые изменения.
- Сохранять обратную совместимость форматов данных и контрактов.
- Учитывать multi-file impact до редактирования.
- Явно отмечать риски и предположения.

## 4. Minimal-diff правила
- Не трогай несвязанные файлы.
- Не переименовывай и не переноси модули без необходимости.
- Не меняй публичные интерфейсы без явной причины и описания последствий.
- Избегай "косметических" массовых правок вместе с функциональными.

## 5. Примеры

### Безопасное изменение (good change)
- Исправить формулу KPI в `backend/analytics/...` и добавить локальную проверку результата.
- Починить обработку пустого входного файла в `backend/ingestion/...` без изменения деплоя.
- Исправить отображение метрики в `dashboard/...` без изменения источников данных.

### Рискованное изменение (risky change)
- Изменить webhook/deploy script “заодно с bugfix”.
- Менять unit/service-файлы `systemd` для ускорения рестарта.
- Перенести вычисления из `backend/analytics/` напрямую в Streamlit-слой.
- Одним коммитом переписать несколько слоев без пошагового плана.
